## K.MOT 動機・世界観

なぜ言語や箇条書きではなく「構造」なのか。RIM が答えようとしている問いの集合。本講義の入り口。

### K.MOT.Structure — 構造で世界は変えられる

RIM が出発点として据える根本主張。**事業・サービス・組織の意思決定は、言葉や箇条書きではなく、関係性の図 (構造) で扱った方が良い**。

リストは「何が大事か」を表現できても、「**どこを動かすと他がどう動くか**」という波及効果を表現できない。意思決定者が「リストから重要なものを選ぶ」モードから「**構造の中でどこを動かすか**」モードへ移ると、施策の打ち方が変わる。

**教え方の勘所**: 講義冒頭で **問い** として開く — 「あなたの組織は、なぜそれが上手く行っているか/上手く行っていないかを、リストではなく図で語れますか?」 受講者は最初詰まる。その詰まり自体が「構造で考えてこなかった」気づきになる。

理論的議論: `principles/core.md` §1

### K.MOT.Cycle — 循環の原則

RIM が選んだ最も決定的な構造の特徴は **「循環がある」** こと。**循環がある構造だけが「上手く行き続ける」**。

直線的因果 (A → B → C で終わり) は外部入力が止まれば止まる。循環構造 (A → B → C → A) は**自分自身が次のサイクルの燃料を生む** — 手を放しても回り続ける自律持続性を持つ。

「うまくいく構造には必ず循環がある」(土屋) は観察結果から導かれた経験則。Value Cycle Model の存在意義そのもの。

**教え方の勘所**: 直線図 (A→B→C) と循環図 (A→B→C→A) を並べて「どちらが持続しますか?」と問う。Amazon の「信頼蓄積 → 想起 → 再利用 → 信頼蓄積」のような身近な例と組み合わせると腑に落ちやすい。

理論的議論: `principles/core.md` §1。実装手順は `K.STG.RsIM.CoreLoop`

### K.MOT.Kakushin — 核心 / 革新 / 確信 (脳内革命の予告)

**RIM が最終的に目指す状態**: 関係者が完成した RIM 図を見たとき、「**あ、本当は我々はこれを目指してたんだ**」という脳内革命を体験する。

「核心 / 革新 / 確信」は 3 つの同音語 (カクシン) で、同じ現象の 3 側面を指す。これは「理解した」(understood) を超え、**「自社の認知が一歩前に進んだ感」(aha experience)** に近い。

RIM 図の成功基準は「整っているか」「ノードが揃っているか」ではなく、**関係者の中で何が起きたか**。

**教え方の勘所**: 第1章では予告のみ。1 スライドで「この講義の到達点は、皆さんが描いた図を見たチームメンバーが『これだったんだ』と腑に落ちる瞬間です」と短く宣言。深掘りは `K.STG.SDIM.Kakushin` (第9章相当) で扱う。

理論的議論: `principles/core.md` §15

### K.MOT.MMA — 総合格闘技時代論

現代ビジネスを「総合格闘技」と表現する時代区分:

| 時代 | 単独で勝てた要素 | 現代における判断 |
|---|---|---|
| 1970-80 年代 | 技術革新単独 (「データを送信する」自体が革新) | 単独では不足 |
| iPhone 期 (2007 以降) | UX 単独 (UX 差別化だけで勝負できた) | 単独では不足 |
| **2020 年代以降** | どれも単独では不足。**UX × Business × Systemic の統合必須** | 必須 |

「総合格闘技」のメタファは、複数視点を **同時に** 戦わせる必要性を強調する。Uber のように「UX で戦いつつ、ドライバー基盤を組み立てつつ、収益機構を回す」三位一体が必要。

**教え方の勘所**: 時系列に 3 つ並べ、各時代の象徴的サービスを当てる (1970s = メインフレーム / iPhone 期 = Facebook・Twitter / 2020s = Uber・Notion)。受講者の所属企業がどの時代の発想で動いているかを当てさせると気づきが起きる。

理論的議論: `principles/core.md` §14

### K.MOT.Era — 「現代において」基準

ノードや矢印の前提を検証する基準: **「その前提は単一視点時代のものか、統合時代に耐えるか」** を問う。

例:
- 「ユーザーは UI が良ければ満足する」 → iPhone 期の前提。2020s では UX + 期待管理 + 信頼蓄積の組合せ必須
- 「いいものを作れば売れる」 → 1970-80s の前提。今は流通・体験・継続性の統合が必須
- 「データを溜めれば差別化になる」 → 2010s 前提。今は AI 活用とセットで初めて差別化

前提が古いと、その上に積み上げた施策ごと無効化される。

**教え方の勘所**: 受講者の自社の主要前提を 3-5 個書き出させ、「それはどの時代の前提ですか?」と演習形式で問う (15 分)。

理論的議論: `principles/core.md` §14

### K.MOT.Uber — Uber ケース (なぜ UX 単独では戦えないか)

Uber を象徴ケースとする理由: **3 視点を「同時に」設計しなければ事業が成立しないことを最も明快に示す**。

- **UX**: 「乗りたい時にすぐ車が来る」「行き先までスムーズ」「決済が摩擦なし」
- **Business**: ドライバー報酬 / プラットフォーム手数料 / 需給に応じた価格 (Surge)
- **Systemic**: マッチングアルゴリズム / 評価システム / ドライバー・乗客の二面市場の自己強化

どれか 1 つが欠けると他の 2 つも崩壊する。UX だけ良くてもドライバーが集まらなければ車が来ない。

**教え方の勘所**: Uber を 3 視点別に書き下し、各視点が他視点に「何を提供している」「何を依存している」を矢印で繋ぐ演習 (15-20 分)。これが Value Cycle の最初の体験になる。

理論的議論: `principles/core.md` §14

## K.THE 理論的系譜

RIM は誰の系譜の上に立つか。Simon → Krippendorff → Design Enablement → Motomura というラインで、半世紀の設計理論の蓄積を吸収している。

### K.THE.Simon — Simon (1969) 設計の定義

Herbert Simon の "The Sciences of the Artificial" (1969) における **設計の操作的定義**: **現在の状態を、好ましい状態に遷移させる行為**。

3 つの含意:
1. **目的指向** — 「好ましい状態」が存在することが前提
2. **状態遷移** — 静的な定義ではなく動的な変換
3. **対象は人工物全般** — 物理製品だけでなく、組織・サービス・規則も含む

RIM はこの定義に立脚しつつ、特に **3 つの視点 (UX/Business/Systemic) の状態遷移を同時に扱う** という拡張を加えている。

**教え方の勘所**: Simon 定義を 1 行で板書し、「設計 = ものづくり、ではない。設計 = **状態を動かすこと**」を強調。これが RIM のすべての出発点。

理論的議論: `principles/theoretical-background.md`

### K.THE.Krippendorff — プロジェクト概念とナラティブ駆動

Klaus Krippendorff の意味論的設計理論 ("The Semantic Turn", 2006) から RIM が受け継ぐもの:

- **プロジェクト (project)**: 設計とは、未来の文脈の中で人工物がどう意味を持つかを構築する行為。「もの作り」ではなく「意味作り」
- **ナラティブ駆動 (narrative-driven)**: 設計は仕様書ではなくナラティブ (物語) によって駆動される。関係者が「これが我々のやりたいことだ」と語れる状態が設計の到達点

RIM の `K.MOT.Kakushin` (脳内革命) と `K.STG.BSM` (ナラティブ変換) は Krippendorff の影響を直接受けている。

理論的議論: `principles/theoretical-background.md`

### K.THE.DesEnable — Design Enablement (FP/SF orbit)

Flying Penguins / Spice Factory 圏で提唱された組織レベルのデザイン推進モデル。**Pulling / Pushing / Demonstrating** の 3 様式で組織にデザイン文化を定着させる:

- **Pulling**: 経営層や事業部門が「デザインを引き出す」状況を作る (需要側)
- **Pushing**: デザイン側から提案を「押し出す」(供給側)
- **Demonstrating**: 実例で「こうなる」を示す (証明)

**RIM は Demonstrating の道具として最も強力** — 図を 1 枚見せることで、口頭説明 100 回より早く「これが我々のやりたいことだ」を共有できる。

理論的議論: `principles/theoretical-background.md`

### K.THE.SecondOrder — Motomura のセカンドオーダーサイバネティクス読解

本村章による RIM の理論的読解。**SDIM (発想/飛躍ステージ) を「セカンドオーダーサイバネティクスの飛躍」として位置付ける**。

セカンドオーダーサイバネティクス (Heinz von Foerster) は **観察者を含めたシステム** を扱う。RIM の文脈では:
- ファーストオーダー (RsIM): 構造を外から記述する
- セカンドオーダー (SDIM): **自分自身が構造の一部であることを認識した上で、ナラティブを再方向化する**

「演繹的発想の壁を超える」現象は、観察者 (設計者自身) のフレームが変わる瞬間として捉えられる。Krippendorff の「ナラティブ再定義」と同じ現象を別言語で記述している (`K.STG.SDIM.FourTraditions` 参照)。

理論的議論: `principles/theoretical-background.md`、`principles/variants.md`

### 演習: 専門分野を系譜に位置づける (15 分)

受講者の専門分野 (例: ソフトウェア工学 / 経営学 / 認知科学) を 4 人の系譜のどこに接続するかを書き出させる。**自分の知識の地図に RIM を位置付ける作業** が、後続の章で参照ポイントを増やす。

## K.NOT 表記法・構造要素

RIM 図を書くための語彙と手順。3 要素 (Actor / Property / Arc) + 記法 + 手順 + NG パターン。

### 3 要素

#### K.NOT.Actor — Actor (S) アクター

RIM 図の 3 要素の 1 つ。**人・組織・擬人化された製品**を表す主体。視覚的には**楕円**で描く。

種類:
- 人: ユーザー / 従業員 / 顧客
- 組織: 自社 / 顧客企業 / パートナー
- 擬人化された製品: 「AI アシスタント」「レコメンダー」など、自律的振る舞いを持つもの

**注意**: Actor は RIM の分析単位**ではない**。分析単位は Property (`K.NOT.Property`)。Actor は Property の所属先として登場する。

理論的議論: `principles/notation.md`

#### K.NOT.Property — Property (O) プロパティ

**RIM の最重要設計判断**: 分析単位を Actor ではなく **Property** に置く。

Property = アクターに付随する**状態属性**。変化する変数として表現される。

例:
- ユーザーが持つ: 期待値 / 満足度 / 信頼
- 組織が持つ: 売上 / 顧客数
- 製品が持つ: 成熟度 / 精度

**なぜ Property を分析単位とするか**:
- 変化は属性の状態変化として起こり、それが隣接 Property に伝搬する (`K.NOT.Arc`)
- Actor ベースだと「人が行動する」レベルで止まり、**循環の駆動力が見えない**
- Property ベースだと「期待値↑ → 行動 → 信頼↑ → 想起↑ → 期待値↑」のような心理駆動循環が描ける

**教え方の勘所**: 受講者は最初「Actor が動く」と書こうとする。「**その Actor の何が変わったの?**」と問い返すと、自然に Property に辿り着く。これが RIM の最大の認知転換ポイント。**「分析単位は Actor ではなく Property」を本章の最重要メッセージとして配置する**。

理論的議論: `principles/notation.md`、`principles/core.md` §6

#### K.NOT.Property.Driver — 駆動プロパティ

循環の**起点**となる Property。微変化が次の行動を引き起こす。

例:
- ユーザー側: 欲求 / 期待値 / 関心 / 不安 (マイナス方向の駆動)
- 事業側: 顧客数 / 投資余力 / ブランド認知
- 製品側: 開発速度 / データ蓄積量

駆動プロパティが見えないと、「**改善の入り口**」が見えない。施策設計のスタート地点はここ。

理論的議論: `principles/notation.md`。心理プロパティの駆動は `K.PSY.Prop.Drv`

#### K.NOT.Property.Result — 結果プロパティ

他の変化の**結果**として動く Property。行動や体験の累積として変動する。

例:
- ユーザー側: 満足度 / 信頼 / 達成感 / 再利用意向
- 事業側: 売上 / 利益率 / 市場シェア
- 製品側: 安定性 / 競争優位 / 技術的負債

**結果プロパティだけを追うと対症療法になる** — 満足度を上げようと UI を改善するが、駆動プロパティの期待値が下がっていれば効かない。

理論的議論: `principles/notation.md`。心理プロパティの結果は `K.PSY.Prop.Res`

#### K.NOT.Arc — Arc (作用)

Property → Property の**伝搬**。`S V O` 形式で記述。

例:
- 「期待値↑ が 検索行動↑ を引き起こす」
- 「満足度↑ が 信頼↑ に作用する」
- 「データ蓄積↑ が 提示精度↑ に作用する」

矢印は方向を持つ。後述の **作用 vs アクション** の区別 (`K.NOT.PropVsAction`) が重要。

理論的議論: `principles/notation.md`

### 記法

#### K.NOT.SHave — `S have O` 記法

**アクターは属性を持つ**ことを示す記法。

例:
- `ユーザー have 期待値`
- `自社 have 顧客数`
- `AI エージェント have 提示精度`

図中の Actor (楕円) と Property (雲型/長方形) を結ぶ所属関係を表す。

#### K.NOT.SVO — `S V O` 記法

**属性が属性に作用する**ことを示す記法。Arc の言語形式。

例:
- `期待値 drives 検索行動` (動詞 V = drives / raises / triggers / affects)
- `満足度 increases 信頼`
- `データ蓄積 improves 提示精度`

英語動詞を使う理由: 日本語の「が」「を」助詞関係だと曖昧になりやすい。**矢印には方向と意味を明示** が原則。

#### K.NOT.PropVsAction — 作用 vs アクション (連続 vs 離散)

**RIM の重要な区別**: 矢印の意味を 2 種類に分ける。

| 概念 | 性質 | 例 |
|---|---|---|
| **作用 (propagation)** | 連続的・自動的 (水が高低差で流れるように) | 「満足度 → 信頼」 (時間をかけて自動的に蓄積) |
| **アクション (discrete action)** | 離散的・意図的 (エネルギーを使う) | 「ユーザーがボタンを押す」 (一瞬の意図的行為) |

**RIM 原則**: **作用 (propagation) を優先的に記述し、アクション (discrete) は最小限に**。離散的アクションだけで埋めると、なぜ循環が回るかが見えない。**連続的作用を見つけることが「循環を発見する」こと**。

**教え方の勘所**: 「水が流れる」(作用) vs 「ボタンを押す」(アクション) を**対比演出**で印象付ける。受講者は最初アクションばかり書く。「その後、自動的に何が起きてますか?」と問うと作用が見えてくる。

理論的議論: `principles/notation.md`

### 手順

#### K.NOT.Standard — 標準型手順 (構造的探索重視)

RIM 図を描く 2 つの手順のうち、**構造的探索を重視する**スタイル。

手順:
1. **Step ①** Actor を列挙し、各 Actor の Property を書き出す (`S have O`)
2. **Step ②** Property 間の作用 (Arc) を書き出す (`S V O`)
3. **Step ③** 各作用の蓋然性チェック (本当に成り立つか — `K.QLY.Plausibility`)
4. **Step ④** 全体を接続し、**複数の循環を発見**する

各ステップで止まって考えるため時間はかかるが、**抜け漏れが少なく、深い構造に到達しやすい**。RIM をしっかり学習したい時、構造の複雑な事業を扱う時に向く。

理論的議論: `principles/notation.md`

#### K.NOT.Simple — シンプル型手順 (ゴール速度重視)

RIM 図を描く 2 つの手順のうち、**ゴール速度を重視する**スタイル。

手順:
1. **Step ①** アクション (「誰の何に X する」) と望ましい状態変化を列挙
2. **Step ②** 接続して循環を発見
3. **Step ③** 蓋然性チェック

⚠ **「誰の何に」を必ず含める** (アクター名 + 属性) — 裸のアクション「データを保存する」だけでは不十分。「**ユーザーの購買履歴を保存する**」と書く。

短時間で 1 枚目を立ち上げたい時、ワークショップ時間が限られる時、初学者の入口として向く。

**演習例 (30 分)**: シンプル型で 15 分ドラフト → 15 分相互共有 → 最頻アンチパターン (`K.NOT.NG.*`) を確認

理論的議論: `principles/notation.md`

### NG パターン (記法レベル)

#### K.NOT.NG.WrongNode — ノード選択誤り

**本質的でないノードを核 (KSF) として配置してしまい、「ad-hoc 理論」化する**。

例: 物流業務の RIM で「配送スピード」をすべての循環の起点にしてしまう。実際は配送スピードは結果プロパティであり、駆動プロパティは「需要予測精度」「ドライバー稼働率」など別にある。

**防ぎ方**: `K.QLY.Granularity` (粒度判断) と `K.QLY.PeerTest` (業界他社識別テスト) を行う。

#### K.NOT.NG.TooFree — 自由すぎて入口不在

**ノードの選び方が自由すぎて、ドメインを知らない読者には入口が分からない**。

例: 業界特殊用語を断りなく使う / 抽象度がバラバラ / 命名規則が章ごとに違う。

**防ぎ方**: 受講者 (または将来の読み手) が「ここから読めば良い」と分かる**起点ノード**を 1 つ明示する。

#### K.NOT.NG.Bloat — 詳細肥大

**詳細を入れすぎて視覚的に重く、全体が把握できなくなる**。

例: 1 つの Actor に 20 個の Property を付ける / 1 つの画面に 50 個のノードを詰める。

**防ぎ方**: `K.LAY.Density` (密度と成熟度) で、成熟度に応じた密度を保つ。**常に 1 画面で視認できる範囲に**。

理論的議論 (3 つすべて): `principles/notation.md`

## K.PER 3視点と Value Cycle

3 つの視点 (UX × Business × Systemic) と、それぞれが回す循環。RIM の構造的本体はここに集約される。

**3 層と循環名の対応 (公用語)**:

| 層 | 英語名 | 日本語名 | 循環 |
|---|---|---|---|
| 1 | UX | UX / 体験 | **UX 満足向上循環** (UX Satisfaction Improvement Cycle) |
| 2 | Business | Business / 事業 | **Business 成長循環** (Business Growth Cycle) |
| 3 | Systemic | システミック / 実装 | **自己強化循環** (Self(-Organization) Reinforcement Cycle) |

各循環名の動詞 (向上 / 成長 / 強化) は意図的に区別している:
- **UX 満足「向上」**: 満足は静止状態ではなく**能動的に高める対象**
- **Business「成長」**: 事業はスケールするもの
- **Systemic「強化」**: 組織は受動的に育つのではなく**能動的に自己強化する**

### K.PER.UX — UX (体験) 視点

#### K.PER.UX.Cycle — UX 満足向上循環

UX 層の Value Cycle 名。**満足を能動的に向上させ続ける**循環構造。

「満足循環」(2026-05-16 時点の旧名) を 2026-05-19 に「満足向上循環」に更新した経緯: UX 満足は獲得して終わりではなく、**継続的に向上させる対象**であることを命名で明示するため。静的な維持 (sustain) ではなく動的な向上 (elevate / improve) を意味する。

理論的議論: `principles/core.md` §20、§8

#### K.PER.UX.Phases — 3 フェーズ整理 (利用前 / 利用中 / 利用後想起)

UX を「一瞬の UI 体験」と捉えるアンチパターンへの対抗概念。UX は 3 つのフェーズに分かれた**ジャーニー**:

| フェーズ | 起きていること | 代表心理プロパティ |
|---|---|---|
| **利用前** | 「使ってみたい」気持ちが立ち上がる | 期待 / 欲求 / 不安 |
| **利用中** | 期待が満たされる / 裏切られる | 達成感 / フラストレーション |
| **利用後想起** | 「またあれ使おう」と思い出される | 信頼 / 記憶の鮮明さ / 想起確率 |

**よくあるアンチパターン**: 多くの Value Cycle が UI (利用中) だけを描いてしまう。結果、「**最初の一回**」しか説明できないループになり、なぜ持続するのかが見えない (`K.ANT.UX` 参照)。

特に **利用後想起** が弱いループはリピートが説明できない。Amazon / YouTube / Uber Eats など持続するサービスはこのフェーズで「想起されやすさ」を作っている (詳細は `K.PSY.PropType.Recall` および `drafts/user_psychology_90min.md` Slide 33)。

理論的議論: `principles/core.md` §8 (UX 用語の理論的バックエンド)

#### K.PER.UX.Design — UX デザイン (心理・行動的循環)

UX 視点で Value Cycle を**実装する**作業。要点は: **教科書的な UX 成功モデルをそのまま採用せず、本ケース固有の循環を構築する** (`K.PER.Crit` 共通方法論)。

具体的には:
- 一般成功モデル (例: AIDMA / 顧客ジャーニー) を**批判的に考察** (必ずしも否定ではない)
- 本ケースの固有要素 (業界・顧客層・チャネル特性) を組み込み
- 利用前/中/後想起の 3 フェーズが**連鎖して**循環するように構築

心理プロパティ深掘りの実践は `K.PSY` ドメインに展開 (駆動 / 結果プロパティ、位置エネルギー比喩、4 つの問いなど)。

### K.PER.BIZ — Business (事業) 視点

#### K.PER.BIZ.Cycle — Business 成長循環

Business 層の Value Cycle 名。事業を「成長」させ続ける循環構造。

#### K.PER.BIZ.Phases — ビジネスフェーズ

Business デザインで考慮する 3 つの基本要素:

- **資金調達** (稼ぎの再投資を含む) — どこからお金を得るか
- **投資対象の選定** — 得たお金をどこに振り分けるか
- **有効なお金の使い方プロセス** — 振り分けたお金を成果に変える運用

これらが連鎖して循環すると Business 成長循環が完成する。

#### K.PER.BIZ.Design — Business デザイン (ビジネス成長循環)

Business 視点で Value Cycle を実装する作業。要点は: **業界の標準ビジネスモデルをそのまま採用せず、本ケース固有の連鎖を構築** (`K.PER.Crit`)。

例:
- 一般 SaaS モデル (MRR × Churn) をベースに、自社の差別化要素 (例: 営業 → カスタマーサクセスの強連結) を組み込む
- 一般小売モデルから出発しつつ、自社の在庫回転戦略を中核に据える

### K.PER.SYS — Systemic (実装) 視点

**Systemic 層だけが 3 要素分解される非対称性**: UX / Business は単一フェーズ系列 (利用前/中/後 や 調達/選定/使い方) で語れるが、Systemic は**異質な 3 要素の組み合わせ**で循環を成立させる。これは 3 視点の中で最も複雑な構造を持つ層。

#### K.PER.SYS.Cycle — 自己強化循環

Systemic 層の Value Cycle 名。**Self(-Organization) Reinforcement Cycle** (自己強化循環)。

命名進化:
- 旧称 (PPTX 公式): "Product Evolution Cycle" — 製品の進化
- 中間 (2026-05-16): "Self(-Organization) Growth Cycle" — 自己 (組織) 成長循環
- 現在 (2026-05-19): **"Self(-Organization) Reinforcement Cycle"** — **強化** を明示

「強化 (Reinforcement)」を選んだ理由: 現代の Systemic 層の主体は**組織それ自体** (人 + システム + AI エージェントの連鎖) であり、受動的に「成長」するのではなく **能動的に自己を強化する**。

理論的議論: `principles/core.md` §20

#### K.PER.SYS.Human — 従業員としての人間

Systemic 層の構成要素 ①。**自律 (非決定論) 的循環**を生む主体。

人間の特徴:
- 判断は文脈依存・非決定論的
- 学習し、過去経験で振る舞いが変わる
- 単体でも循環を持つ (例: 経験 → 自信 → 挑戦 → 経験)

「何を人間に任せ、何を任せないか」を厳密に定義することが Systemic デザインの中核 (`K.PER.SYS.Design`)。

#### K.PER.SYS.IT — 会社が提供する IT システム

Systemic 層の構成要素 ②。**アルゴリズム (フローチャート / 決定論) 的フロー**。

IT システムの特徴:
- 入力 → 出力が決定論的
- 自律性は持たない (チューリングマシン + データベース)
- **単体では循環しない** — 人間 (Human) や AI エージェント (AI) と組み合わさって初めて循環の一部になる

「システムは何を可視化するか」を定義することが設計の要点。

#### K.PER.SYS.AI — 疑似的自主性をもつ AI エージェント

Systemic 層の構成要素 ③。**人間とシステムの中間的挙動**。**LLM + ハーネスの設計論**。

AI エージェントの特徴:
- 完全な自律ではないが、決定論的でもない (中間)
- 状況に応じて応答を変える「疑似的自主性」
- ユーザー能動型 (ユーザーが使う) と エージェント・プル型 (AI が主体的に取得・行動) の 2 モードがある

設計時には**両モードを考慮**することが必要 (`principles/core.md` §6 参照)。

#### K.PER.SYS.Design — Systemic デザイン

Systemic 視点で Value Cycle を実装する作業。要点は: **共通フェーズは存在しない**。

①人間の自律的循環 + ②IT システムの決定論的フロー + ③AI エージェントの中間的挙動 — この**異質な 3 要素を組み合わせて循環を成立させる設計論**。

UX / Business の Design とは違い、ベースとなる標準フェーズが無いため、**ケースごとに 3 要素の組み合わせ方を設計し直す**ことになる。

### K.PER 共通

#### K.PER.Crit — 「批判的考察 → 本ケース固有の連鎖」共通方法論

3 層 (UX/Business/Systemic) すべてに共通する設計姿勢。**独立した方法論として宣言する**。

原則:
- 各層には標準的な成功モデル / 成功メタファーが存在する (AIDMA / SaaS モデル / プラットフォーム理論 など)
- それを**そのまま採用しない**
- **批判的に考察 (必ずしも否定ではない) し、本事業ケースならではの連鎖を作る**

なぜこの姿勢が必要か: 標準モデルそのままだと業界他社と区別がつかなくなる (`K.QLY.PeerTest` 参照)。RIM 図の差別化は、各層のベースモデルを本ケースに合わせ**再構成する**ことから生まれる。

**教え方の勘所**: 第5章で 1 セクション (15 分) を割いて**独立宣言**する価値がある。3 層詳細に入る前に、この共通方法論を提示しておくと、各層の Design が「**標準モデル + 本ケース固有の再構成**」という一貫構造で理解できる。

#### K.PER.Exchange — 三層交換構造

各層で「何と何が交換されているか」を整理した枠組み:

| 層 | 交換される本質 |
|---|---|
| **Market** | 価値 ⇄ 収益 / 信頼 / ブランド / シェア |
| **Service** | 情報 ⇄ リソース (時間・金・注意・行動) |
| **Organization** | 役割・責任 ⇄ 報酬・評価・依存 |

これは 3 層に対応するが、**交換** (何が流れているか) を強調する視点。RIM 図のアクター間関係を整理する補助フレームワーク。

理論的議論: `principles/notation.md` (三層交換構造)

## K.STG 3 ステージ (RsIM / SDIM / BSM)

RIM の運用フェーズ構造。**RsIM (構造を作る) → SDIM (構造を超える) → BSM (構造を伝える)** という流れで、それぞれ異なる作業質を持つ。

### K.STG.Overview — 3 ステージ全体像

RIM は 3 ステージで構成される。**RsIM** (分析 / 発想) → **SDIM** (発想 / 飛躍) → **BSM** (伝達)。3 ステージはどれも欠けては成立しない:

| ステージ | 何をするか | 性質 | 本講義での位置 |
|---|---|---|---|
| **RsIM** | 構造を作る | 連続的イノベーション探索 (AI 支援可能) | 第8章 — **本講義の主戦場** |
| **SDIM** | 構造を超える | 現象 (メソッドではない) | 第9章 |
| **BSM** | 構造を伝える | ナラティブ変換 | 第10章 (概論レベル) |

3 ステージは**直線的に進む**わけではない。RsIM の積み上げから SDIM が立ち上がり、SDIM の成果が BSM で伝達される、という階層的順序。

**演繹的発想の壁**: RsIM の積み上げだけでは到達できない領域がある (`K.STG.SDIM.Wall`)。そこで SDIM が起きる。

理論的議論: `principles/three-stages.md`、`principles/core.md` §19

### K.STG.StanceBase — 「派手さ」と「姿勢のベース」

3 ステージの中で「見せ場」と「ベース」が乖離する構造的事実:

- **見せ場 (派手)**: 第9章の SDIM (脳内革命) と 第10章の BSM (経営伝達) — 講義のクライマックスに見える
- **姿勢のベース (地味)**: 第8章の RsIM — **すべてを成立させる土台**

**SDIM の飛躍は、RsIM 姿勢を持続させた者にのみ立ち上がる**。BSM の伝達は、RsIM で作った構造があって初めて意味を持つ。

**教え方の勘所**: 第7章 (全体像) の最後に**明示的に宣言する**: 「派手なのは第9章 (SDIM) と第10章 (BSM) に見えるが、それらすべてを成立させる**ワーキング姿勢のベースは第8章 RsIM にある**。地味だが、これが核」。この宣言があると、第8章の重さが伝わる。

### K.STG.RsIM — RsIM (分析 / 発想)

**RIM 実践者が日常的に取るワーキング姿勢であり、同時に手順**。**中核循環は、その姿勢が外から見えた時の形**。

#### K.STG.RsIM.Stance — RIM 実践者のワーキング姿勢

RsIM の核心は「メソッド (手順)」ではなく「**姿勢 (stance)**」として理解した方が伝わる。手順はあくまで姿勢が外形化したもの。

姿勢は 4 つの要素から成る (次の 4 サブノード):

##### K.STG.RsIM.Stance.Plausibility — 蓋然性を常に疑う

自分の書いた図 (モデル) の各ノード・各矢印について、「**本当にそれは成り立っているか?**」を常に疑う姿勢。

「業界の常識だから」「皆そう言うから」では蓋然性チェックを通過しない。自分が描いたモデルの中で、**最も怪しいノード/アーク**を見つける目を持つ。

これが姿勢の起点 — 怪しさが見つからない人は次の「視点切替」に進めない。

##### K.STG.RsIM.Stance.Switch — 視点を躊躇なく切り替える

怪しいノード/アークが見つかったら、**異なる視点 (UX / Business / Systemic) で別モデルを書く**。

「自分は UX 専門だから」「Business は他部署の仕事」という縄張り意識を捨て、必要に応じて UX ↔ Business ↔ Systemic を往復する。これが「**Recursive** Innovation Method」の Recursive (再帰的) の意味。

##### K.STG.RsIM.Stance.Integrate — 別視点モデルを元モデルに統合する

別視点で書いたモデルを、**元のモデルに重ね合わせて統合する**。単に並列に並べるのではなく、**新しい Value Cycle Model** として再構成する。

統合の結果、新しい循環が見えるか、もしくは元の循環の蓋然性がより強固になる。

##### K.STG.RsIM.Stance.Asymptotic — 漸近的・持続的に改善する

**「一発で正解を出さない」姿勢**。RIM は一回の作業で完成するものではなく、**漸近的 (asymptotic)** に正解に近づいていく。

土屋が「RIM で最も学んでほしいもの」と表現するのが、まさにこの**漸近的・持続的改善アプローチ**。完璧主義を捨て、現時点のベストを書き、次のイテレーションで改善する。

#### K.STG.RsIM.CoreLoop — 中核循環

上記 4 つの姿勢を持続させると、自然に立ち上がってくる作業循環:

```
初期モデル
  → 蓋然性が怪しいノード/アーク を検出  (Stance.Plausibility)
  → 視点切替で別モデル記述  (Stance.Switch)
  → 統合してバリューサイクルモデルへ  (Stance.Integrate)
  → 漸近的改善  (Stance.Asymptotic)
  → 循環 (元に戻る)
```

これが「Core Working Loop」(中核循環)。RIM をやるとはこのループを回し続けること。**姿勢を持たずにこのループを「形だけ」回そうとしても回らない**。

**教え方の勘所**: 章の前半で**姿勢の 4 要素を先に提示**し、中盤で「これらを持続させると、外から見ると自然にこのループになる」として中核循環を提示する順序が効果的。**逆順 (ループを先に提示) だと、メカニカルな手順に見えてしまい、姿勢が伝わらない**。

理論的議論: `principles/core.md` §10

#### K.STG.RsIM.AnyStart — 「出発点はどこでもよい」

**RsIM の決定的特徴**: 出発点は UX でも Business でも Systemic でも、どこからでも良い。**「UX リサーチから始めるべき」というデザインシンキング正統派の教義を否定する**。

ただし常識的には:
- **Business 起点が問いを生みやすい** (事業構造の検討から「ユーザーは本当にそうするか?」という UX の問いが自然に立ち上がる)
- UX 起点は、UX チームが強い場合は有効
- Systemic 起点は、技術的制約が支配的な場合に有効

抽象度のレベルでも起点は変動する。**「正しい出発点」という幻想を持たない**ことが重要。

**アンチパターン**: 課題仮説なしのユーザーリサーチ先行 (`K.ANT.Process.ResearchFirst`)。「まずユーザーに聞こう」は、何を解こうとしているのか分からない状態でユーザーを巻き込むため、有用な答えが得られない。

理論的議論: `principles/core.md` §11

### K.STG.SDIM — SDIM (発想 / 飛躍)

RsIM の積み上げの上に立ち上がる**飛躍的イノベーション現象**。第8章で作った RsIM 姿勢が前提。

#### K.STG.SDIM.Phenomenon — SDIM は現象であってメソッドではない

**最も重要な前提**: SDIM は**現象**であり、**メソッドではない**。

「SDIM の手順は?」と聞かれても答えはない。SDIM は手順を踏めば起きるものではなく、**RsIM 姿勢を持続させた人にのみ立ち上がる体験**。

AI もメソッドとして提供できない。AI ができるのは**構造的条件を整える**こと (`K.AIC.RevelationCondition`) — SDIM そのものを起こすことではない。

理論的議論: `principles/three-stages.md`

#### K.STG.SDIM.Wall — 演繹的発想の壁

**RsIM の積み上げだけでは到達できない領域**が存在する。これを「演繹的発想の壁」と呼ぶ。

演繹的思考 (前提 → 結論) を積み重ねても、**前提のフレーム自体を変えること**はできない。フレームを変える跳躍が SDIM の本質。

例: 「タクシー会社は車両を所有する」という前提を演繹的に最適化していくと、より効率的なタクシー会社にはなれるが、Uber にはなれない。Uber は「車両を所有しない」と前提自体を変えた。これが演繹的発想の壁を超える例。

理論的議論: `principles/three-stages.md`、RIM 公式 PDF Page 9

#### K.STG.SDIM.Kakushin — 脳内革命 / 核心 / カクシン

SDIM 現象の中身: **関係者の脳内で起きる「あ、本当は我々はこれを目指してたんだ」という革命的気づき**。

3 つの同音語 (カクシン) で同じ現象の 3 側面を指す:
- **核心** — 自社の本質の中心が見えた
- **革新** — 視野が更新された (innovation の文字通りの意味)
- **確信** — 「これだ」という強い納得

これは「理解した」(understood) を超え、**「自社の認知が一歩前に進んだ感」(aha experience)** に近い felt experience。

`K.MOT.Kakushin` (第1章での予告) の本格的展開がここ。

理論的議論: `principles/core.md` §15

#### K.STG.SDIM.Analogy — アナロジー思考

**演繹的発想の壁を超える触媒の 1 つ**: 他領域からの構造的類比。

例: 「自動車業界に Toyota 方式 (リーン生産) を持ち込む」のではなく、「**Toyota 方式の構造を別の業界 (例: ソフトウェア開発) に持ち込むと何が起きるか?**」と類比的に問う。

アナロジー思考は触媒であって生成器ではない。アナロジーを使えば必ず飛躍が起きるわけではない。

#### K.STG.SDIM.Inclusive — インクルーシブ思考

**演繹的発想の壁を超える触媒の 1 つ**: 視点を包摂する統合的視座。

複数視点 (UX / Business / Systemic / ステークホルダー / 時間軸 など) を**対立軸ではなく包摂的に捉える**思考様式。「どちらか一方」ではなく「両方を含む上位概念は何か?」と問う。

例: 「効率 vs 多様性」と二項対立にせず、「効率を生む多様性とは何か?」と包摂的に問い直す。

#### K.STG.SDIM.LastMile — ラストワンマイル原則

**SDIM の飛躍は最後の一歩を人間に残す**。

AI も他の人間 (本人以外) も、本人の脳内で起きる脳内革命を代行することはできない。RsIM までの構造的支援は AI 可能、SDIM の飛躍は人間に残す — これが rim-coach の設計思想の中核 (`K.AIC.LastMile`)。

理論的議論: `principles/core.md` §12

#### K.STG.SDIM.FourTraditions — 同一現象の 4 つの呼び名

SDIM (= 演繹的発想の壁を超える現象) は、複数の伝統で別言語で記述されてきた:

| 伝統 | 呼び名 |
|---|---|
| 土屋 | **脳内革命 / 核心 / カクシン** |
| 本村 | **セカンドオーダーサイバネティクス的 飛躍** |
| Krippendorff | **ナラティブの方向再定義** |
| RIM 公式 | **演繹的発想の壁を超える / 飛躍的イノベーション** |

これらは**同じ現象を別言語で記述している**。受講者の背景によって、どの呼び名が最も腑に落ちるかは違う。

理論的議論: `principles/three-stages.md`、`principles/theoretical-background.md`

### K.STG.BSM — BSM (伝達)

完成した RIM 図を、関係者 (経営層・投資家・社内ステークホルダー) に**伝わる言語**に変換するステージ。

#### K.STG.BSM.Purpose — 経営層・投資家・関係者への伝わる言語への変換

**BSM の目的**: RIM 図 (構造) は描いた本人と RIM を学んだ人にしか正しく読めない。それを**読んでない人にも伝わる物語 (narrative)** に変換することが BSM。

構造設計 (RIM) と伝達 (BSM) は**別の能力**。図が美しくても伝わらなければ脳内革命は起きない。逆に、図がシンプルでも伝達が巧みなら関係者が動く。

理論的議論: `principles/three-stages.md`

#### K.STG.BSM.FiveElements — BSM の 5 要素

完成した RIM 図をナラティブに変換する際の標準フレーム:

1. **何を提供するか** (What)
2. **どう提供するか** (How)
3. **どんな価値をもたらすか** (Value)
4. **なぜ取り組むのか** (Why — MVV 接続)
5. **どうやって永続化するか** (Sustainability)

5 要素は順序ではなく**チェックリスト**。実際のナラティブの構成順序はケースによる。

#### K.STG.BSM.LLMAffinity — LLM 親和性

**ナラティブ生成は LLM の得意領域**。構造化された RIM 図 (5 要素を満たす) を入力として与えれば、質の高い BSM ドラフトを LLM が生成できる。

ただし最終的なナラティブの選定・調整は人間が行う (ターゲット聴衆との文脈調整、トーンの調整)。AI 親和性が高いことと AI が全部やるべきは別問題。

#### K.STG.BSM.Scope — rim-coach 対象外、将来 `rim-bsm` 別スキル化

**rim-coach OSS は BSM を対象外**としている。理由:
- rim-coach は RsIM (構造的探索) を中核に据えている
- BSM はナラティブ生成という別の作業質
- 将来的に独立した `rim-bsm` スキル/プロダクトとして切り出す計画

本講義では概論レベル (1h) のみ扱う。実務的に BSM を深く使いたい場合は `rim-bsm` の将来リリースを待つか、別途プロンプト設計が必要。

## K.PSY ユーザー心理プロパティ
