## Part I. 「なぜ RIM か」— 動機と世界観 (2.5h)

### 第1章 RIM とは何か — 構造で世界は変えられる (45分)
- **問い**: なぜ言葉や箇条書きではなく「構造 (関係性の図)」なのか
- **キーコンセプト**: バリュー・チェーン / キモ / 循環 (循環がある構造だけが「上手く行き続ける」)
- **核心の予告**: 「あ、本当は我々はこれを目指してたんだ」= 脳内革命 (§15 の予告)
- **依存**: なし (起点)

### 第2章 現代ビジネスは「総合格闘技」である (45分)
- **時代区分**: 1970-80 年代 (技術単独) → iPhone 期 (UX 単独) → 2020 年代 (UX × Business × Systemic の統合必須)
- **Uber ケース**: なぜ UX だけでは戦えないか
- **「現代において」基準**: 前提が旧時代の単一視点のものか、統合時代に耐えるか
- **依存**: 第1章

### 第3章 RIM の理論的系譜 (1h)
- Simon (1969): 設計 = 状態遷移
- Krippendorff: プロジェクト概念とナラティブ駆動
- Design Enablement (FP/SF系): Pulling / Pushing / Demonstrating
- Motomura のセカンドオーダーサイバネティクス読解
- **演習**: 自分の専門分野の「設計」観をこの系譜の中に位置づける (15分)
- **依存**: 第1章

## Part II. 表記法と基本構造 (3h)

### 第4章 RIM の3要素 — Actor / Property / Arc (1h)
- **Actor (S)**: 人・組織・擬人化された製品。楕円で表す
- **Property (O)**: アクターに付随する状態属性。**RIM の分析単位は Actor ではなく Property**
- **Arc (作用)**: Property → Property の伝搬。`S V O`
- **作用 vs アクション**: 連続的・自動的 (水が流れる) vs 離散的・意図的 (ボタンを押す)
- **記法**: `S have O` (アクターは属性を持つ) / `S V O` (属性が属性に作用)
- **依存**: 第1章

### 第5章 3視点と3つの Value Cycle (1h)

**3層と循環 (Value Cycle) の対応**
- **UX (体験)** — UX 満足向上循環
- **Business (事業)** — Business 成長循環
- **Systemic (実装)** — 自己 (組織) 強化循環 ※旧称: Product Evolution Cycle

**Systemic 層だけが 3 要素に分解される** (UX / Business は単一フェーズ系列で語れるが、Systemic は異質な 3 要素の組み合わせ):
- ① **従業員としての人間** — 自律 (非決定論) 的循環
- ② **会社が提供する IT システム** — アルゴリズム (フローチャート / 決定論) 的フロー。※単体では循環しない
- ③ **疑似的自主性をもつ AI エージェント** — 人間とシステムの中間的挙動をする LLM + ハーネスの設計論

**各層の「デザイン」共通方法論**: 各層には標準的な成功モデル / 成功メタファーが存在するが、それをそのまま採用せず、**批判的に考察 (必ずしも否定ではない) し、本事業ケースならではの連鎖を作る**。

- **UX デザイン (心理・行動的循環)**: 利用前 / 利用中 / 利用後想起 — UI だけではない。これらが連鎖するよう、本ケース固有の循環を構築
- **Business デザイン (ビジネス成長循環)**: 資金調達 (稼ぎの再投資を含む) / 投資対象の選定 / 有効なお金の使い方プロセス。これらが連鎖するよう、本ケース固有の循環を構築
- **Systemic デザイン (自己強化循環)**: 共通フェーズは存在しない。①人間の自律的循環 + ②IT システムの決定論的フロー + ③AI エージェントの中間的挙動 — この異質な 3 要素を組み合わせて循環を成立させる設計論

- **依存**: 第4章


### 第6章 描画手順 — 標準型 vs シンプル型 (1h)
- **標準型 (構造的な探索重視)**: Actor 列挙 → Property → 作用 → 蓋然性チェック → 複数の循環の発見
- **シンプル型 (ゴールへのスピード重視)**: 「誰の何に X する」アクション列挙 + 望ましい状態変化の列挙 → 接続 → 循環の発見 → 蓋然性チェック
- **どちらを選ぶか**: 抽象度・プロジェクト成熟度・チーム熟練度で選択
- **演習**: シンプル型で簡易 RIM を 15 分で描く (30分)
- **依存**: 第4・5章

## Part III. 3 ステージ詳細 — RsIM / SDIM / BSM (4h)

### 第7章 3 ステージ全体像 (30分)
- RIM は 3 ステージで構成される: **RsIM (分析 / 発想)** → **SDIM (発想 / 飛躍)** → **BSM (伝達)**
- **位置づけ**:
  - **RsIM**: 連続的イノベーション探索ゾーン (AI 支援可能領域、本講義の主戦場)
  - **SDIM**: **現象であってメソッドではない**。脳内革命 / 核心 / カクシン (RsIM の延長線上で立ち上がる飛躍)
  - **BSM**: 関係者・経営層・投資家への**ナラティブ変換**ステージ
- **演繹的発想の壁**: RsIM の積み上げだけでは到達できない領域がある → そこで SDIM が起きる
- **ステージ間の橋渡し**: RsIM (構造を作る) → SDIM (構造を超える) → BSM (構造を伝える)。3 ステージはどれも欠けては成立しない
- **派手さと姿勢のベース**: 講義の見せ場は第9章 (SDIM = 脳内革命) や第10章 (BSM = 経営伝達) に見えがちだが、**それらすべてを成立させる「ワーキング姿勢」のベースは第8章 RsIM にある**。地味だが、これが核
- **本講義での扱い**: 各ステージを独立章で詳細展開 (第8章 = RsIM / 第9章 = SDIM / 第10章 = BSM)
- **依存**: 第5・6章

### 第8章 RsIM — RIM 実践者のワーキング姿勢 (1.5h)
- **位置づけ**: RsIM は「メソッド (手順) であると同時に、RIM 実践者が日常的に取る**ワーキング姿勢**」である。中核循環はその姿勢が外から見えた時の形
- **姿勢の 4 要素**:
  - 自分のモデルの**蓋然性を常に疑う**
  - **視点を躊躇なく切り替える** (UX / Business / Systemic 間を往復)
  - 別視点で書いたモデルを**元モデルに統合する**
  - **漸近的・持続的に改善する** (一発で正解を出さない)
- **中核循環 (姿勢が外形化したもの)**:
  ```
  初期モデル → 蓋然性が怪しいノード / アーク 検出 → 視点切替で別モデル記述
  → 統合してバリューサイクルモデルへ → 漸近的改善 → 循環
  ```
- **「漸近的・持続的改善アプローチ」= 土屋が最も学んで欲しいもの**
- **「出発点はどこでもよい」**: UX 先行教義の否定。ただし常識的には Business 起点が問いを生みやすい
- **アンチパターン (姿勢が崩れる典型)**: 課題仮説なしのユーザーリサーチ先行 / 一発正解を狙う / 視点切替をサボる
- **依存**: 第7章

### 第9章 SDIM — 演繹的発想の壁と発火条件 (1h)
- **SDIM の本質**: 現象であってメソッドではない。**脳内革命 / 核心 / カクシン** (3 つの同音語)
- **発火条件 (AI が直接起こせない領域での触媒)**:
  - **アナロジー思考**: 他領域からの構造的類比
  - **インクルーシブ思考**: 視点を包摂する統合的視座
- **AI の役割は構造的条件を整えること**: ラストワンマイル原則 — 飛躍そのものは人間に残す
- **4 つの伝統が同じ現象を指す**: 土屋「脳内革命」/ 本村「セカンドオーダー飛躍」/ Krippendorff「ナラティブ再定義」/ RIM 公式「演繹的発想の壁を超える」
- **第8章との関係**: SDIM は RsIM 姿勢を持続させた者にのみ立ち上がる現象。姿勢が無ければ壁にすら近づけない
- **依存**: 第7・8章

### 第10章 BSM — 経営伝達ナラティブ (1h)
- **BSM の目的**: 完成した RIM 図を、経営層・投資家・関係者に**伝わる言語**に変換する
- **5 要素**: 何を提供するか / どう提供するか / どんな価値をもたらすか / なぜ取り組むのか (MVV 接続) / どうやって永続化するか
- **RIM 図との関係**: 構造設計 (RIM) と伝達 (BSM) は別の能力。図が美しくても伝わらなければ脳内革命は起きない
- **AI 親和性が高い領域**: ナラティブ生成は LLM 得意領域。構造化された RIM 図を入力にすれば質の高いドラフトが出やすい
- **講義での扱い**: 概論レベル。本格展開は将来 `rim-bsm` 別スキルで
- **依存**: 第7・8章

## Part IV. 品質・粒度・レイアウト (3h)

### 第11章 粒度 = 抽象度 = 競合差別化 (1h)
- **理想**: 「自社の競合差別化された強みが表現される範囲で、もっとも抽象的な状態」
- **3 つのテスト**:
  - **粗すぎる**: 業界他社と区別不能 (楽天 vs Amazon が同じ RIM になる) = NG
  - **細かすぎる**: 行動の連鎖 / 機能の羅列 / UI 操作列挙 = 細分化の罠
  - **混在 (最も難しい)**: MECE でない、属性が突然遠くに出現、文章として自然な日本語にならない
- **§7・§16 統一**: 粒度 = 抽象度 = 本質判断は同じ次元
- **依存**: 第5章

### 第12章 蓋然性チェックと品質チェック (1h)
- パターン認識による違和感の検出
- 5 つの蓋然性チェック技法
- ミルフィーユフロー: AI で構造化 → 人間で整える → AI 補完 → 人間再修正
- **エキスパート相談規律**: AI は判断を下さず、専門家相談を促す
- **依存**: 第8・11章

### 第13章 レイアウト原則 (1h)
- **XY 軸意味論**: X = 時間 / 流れ、Y上 = 抽象度高 / 原因側
- **循環可視性の決定的慣習**: 太線 + 暖色 (オレンジ・赤) で目立たせる / 背景因果は細線 + 寒色
- **循環形状は二次的** (可視性 > 形状)
- **密度は成熟度で変わる**: 試行錯誤期はスパース / 仕上げ期はタイト / 常に1画面
- **シンメトリーは不要、ただし業界パターン類似は理解を助ける**
- **ツール選定**: Miro 推奨 / FigJam 可 / Mermaid は構造列挙のみ
- **依存**: 第4章 (Actor / Property 理解)

## Part V. 山脈・峡谷・変形 (2h)

### 第14章 マウンテンレンジ & 峡谷モデル (1h)
- **「ネガティブ」の正体は峡谷 (谷)**: positive → positive → negative の negative とは、2 つの強み (山頂) を繋ぐべき欠落した接続
- **イノベーション = 峡谷を見つけて橋を架けること**
- **漸近的イノベーションの本質**: 山と山の間の落差を構造化する
- **依存**: 第7・8章

### 第15章 RIM の 2 つの流派 — Tsuchiya 式 vs Motomura 帰納式 (45分)
- **Tsuchiya 式**: 山脈 + 峡谷 (positive 循環起点)
- **Motomura 帰納式**: ネガティブサイクル → 反転点探索 (オセロ的反転)
- **共存と使い分け**: 漸近的か飛躍的か、組織の現状理解度で選ぶ
- **依存**: 第14章

### 第16章 適用境界 — 意志統一 vs 解釈発散 (15分)
- **RIM の本領**: 関係者の意志統一 (MVV も含む)
- **言語が向く領域**: 解釈の余白を残したい時、ブランドの look & feel
- **ハイブリッド**: ブランドガイドラインなど運用ルールは階層構造 (RIM 的) で OK
- **誤解の修正**: §5の「MVV は言語」は粗い表現。**主題ではなく目的で線を引く**
- **依存**: 第1・3章

## Part VI. アンチパターンと AI 協働 (1.5h)

### 第17章 アンチパターン総覧 (45分)
- Business 系: ユーザー行動の羅列、システム挙動の羅列
- UX 系: UI / 利用中だけの UX、心理的ジャーニー不在
- Systemic 系: 人 / システム / AI の役割境界不在
- プロセス系: ユーザーリサーチ先行、課題仮説不在
- 循環構造系: 循環が見えない、循環していない静的構造
- AI コーチング系: AI が抽象度を判断する、AI が飛躍を生む
- 実践者マインドセット: 完璧主義、ツール先行
- **依存**: 第8・11章

### 第18章 AI 協働の規律 (45分)
- **§9 ヒント、答えではない**: AI は抽象度を判定せず、本人に問い返す
- **§12 ラストワンマイル**: 飛躍は人間の領域
- **§15 核心**: AI は脳内革命の「条件を整える」だけ
- **ミルフィーユパターン**: AI 構造化 ↔ 人間レイアウト ↔ AI 補完 ↔ 人間判断
- **rim-coach の設計思想**: バリデーター / ジェネレーターではなく、問いを返すコーチ
- **依存**: 第12・13章

## Part VII. 実践・演習・キャップストーン (5h)

### 第19章 ケーススタディ (1.5h)
- 公開素材ベースの RIM 図 2-3 件を題材に
  - 良い例: バリューサイクルが見える、循環が太線で目立つ、粒度が差別化を表現
  - 悪い例 (典型的失敗): 行動列挙、UI 偏重 UX、Systemic 不在
- **依存**: 全章

### 第20章 演習1 — シンプル型で描いてみる (1.5h)
- 受講者の所属組織 / 関心領域で、シンプル型で 30 分ドラフト → 相互レビュー → 蓋然性チェック
- **依存**: 第6・11・12章

### 第21章 演習2 — キャップストーン (2h)
- 演習1のドラフトを標準型に昇格 → 視点切替 → 統合 → 循環発見 → レイアウト
- 関係者プレゼンを想定したナラティブ簡易作成 (BSM の入り口)
- **発表 & 評価軸**: 「見た人に脳内革命が起きそうか」を相互フィードバック
- **依存**: 全章
